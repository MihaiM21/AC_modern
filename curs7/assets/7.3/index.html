7.3 INTERDEPENDENȚA SET DE INSTRUCȚIUNI - ORGANIZARE INTERNĂ

Există trei tipuri principale de arhitecturi bazate pe locul de stocare a operanzilor: arhitectură pe bază de acumulator, arhitectură pe bază de registre de uz general și arhitectură pe bază de stivă.

Tipuri de arhitecturi


ARHITECTURĂ PE BAZĂ DE ACUMULATOR Ac

În arhitectura pe bază de acumulator, unul dintre operanzi este implicit acumulatorul, iar celălalt operand este specificat în instrucțiune.


Exemplu de instrucțiuni:

  • add x - Ac ← Ac + x sau Ac ← Ac + @x
    adună la acumulator conținutul registrului x sau al memoriei de la adresa dată de x

  • Avantaje
    • Instrucțiuni scurte (acumulatorul este referit implicit)
    • Densitate relativ mare a codului
  • Dezavantaje
    • Acumulatorul constituie un puternic factor de limitare al vitezei de lucru al procesorului

ARHITECTURĂ PE BAZĂ DE REGISTRE DE UZ GENERAL RG

În arhitectura pe bază de registre de uz general, toți operanzii sunt specificați explicit în instrucțiune, fie din registre, fie din memorie.


Exemple de instrucțiuni:

  • add r1,r2,r3 - r1 ← r2 + r3
  • add r1,r2 - r1 ← r1 + r2 sau r2 ← r1 + r2
  • mov r1,r2 - r1 ← r2 sau invers
  • add r1,x - r1 ← r1 + @x

  • Avantaje
    • Ușurință în generarea codului de către programe compilatoare
    • Rezultatele intermediare pot fi stocate în registre (foarte rapid)
    • Dacă nu ar exista aceste registre, foarte multe din rezultatele intermediare ar fi trebuit stocate în memoria principală ceea ce ar fi micșorat viteza de prelucrare
  • Dezavantaje
    • Fiecare registru trebuie adresat de către instrucțiunea ce-l utilizează, astfel că lungimea instrucțiunilor este relativ mare

Subtipuri:

  • Instrucțiuni cu 3 adrese (RISC)

    • La acest tip de arhitecturi se folosesc doar instrucțiuni de load (încarcă) și store (stochează) când se comunică cu memoria, iar toate celelalte instrucțiuni se referă la operanzi din registrele UCP (deci prelucrare foarte rapidă).


    Exemplu pentru x = a × (b + c):

    load r1,a    ; r1 ← @a
load r2,b    ; r2 ← @b
load r3,c    ; r3 ← @c
add r2,r2,r3 ; r2 ← (r2 + r3)
mul r1,r1,r2 ; r1 ← (r1 * r2)
store x,r1   ; @x ← r1
  • Instrucțiuni cu 2-3 adrese (CISC)

    • Instrucțiunile pot efectua operații aritmetice sau logice cu operanzi din memorie.


    Exemplu pentru x = a × (b + c):

    Cu 3 adrese:

    add r1,b,c   ; r1 ← @b + @c
mul x,r1,a   ; @x ← r1 * @a

    Cu 2 adrese:

    load r1,b    ; r1 ← @b
add r1,c     ; r1 ← r1 + @c
mul r1,a     ; r1 ← r1 * @a
store x,r1   ; @x ← r1

ARHITECTURĂ PE BAZĂ DE STIVĂ ST

În arhitectura pe bază de stivă, operanzii sunt stocați într-o stivă, iar operațiile se efectuează pe elementele din vârful stivei.


Exemple de instrucțiuni:

  • push x - trimite cuvântul de adresă x în vârful stivei, VS ← @x
  • add - aduce primele 2 articole din stivă, le adună și stochează rezultatul în stivă (zero adrese)
  • mul - aduce primele 2 articole din stivă, le înmulțește și stochează rezultatul în stivă (zero adrese)
  • pop x - @x ← VS, VS pointează la adr+1

  • Exemplu pentru x = a × (b + c) 
    • push a      ; vs ← @a, fie „adr" adresa curentă a vârfului stivei (vs)
            ; iar stiva crește spre adrese mici
push b      ; vs ← @b, la adr-1
push c      ; vs ← @c, la adr-2
add         ; vs ← (b + c), stocat la adr-1 (zero adrese)
mul         ; vs ← a*(b+c), stocat la adr (zero adrese)
pop x       ; @x ← vs, vs pointează la adr+1
  • Notația poloneză (inversă)

    • Pentru a evalua expresiile aritmetice într-un calculator stivă, e necesar să se convertească expresia în notația poloneză (inversă). Ideea a pornit de la matematicianul polonez Lukasievics J, care a notat o expresie de forma "a + b" ca "ab+".


    Exemple de notație poloneză:

    • x = a × (b + c) = abc+×
    • x = (a + b) × (c + d) = ab+cd+×
    • x = (a + b) × (c + d) × (e + f) = ab+cd+ef+××

    După cum se vede, notația poloneză are avantajul că nu folosește paranteze. Ba mai mult, pornind de la stânga la dreapta se indică toate operațiile ce trebuie efectuate, în ordine, pe o arhitectură de tip stivă.

  • Avantaje
    • Evaluare extrem de ușoară a operațiilor pe baza notației inverse
    • Instrucțiuni cu lungime foarte redusă
  • Dezavantaje
    • O stivă este mai dificil de adresat în mod aleator
    • Codul rezultat este relativ ineficient
    • Stiva introduce și o limitare în viteză
    • De aceea acest tip de arhitectură este rar implementat la procesoare de uz general
